TESTING STRATEGY FOR WATCHDOG:

Phase 1: Unit Tests (No Dependencies)
  - Test health check logic in isolation
  - Test heartbeat message construction
  - Test state tracking without device interaction
  - Use lightweight harness classes
  - Files: core/tests/device_bootstrap_test.py pattern

Phase 2: Integration Tests (Simulation Only)
  - Test health checks with simulated devices
  - Inject timeout/disconnect faults
  - Verify recovery logic with fault injection
  - Use SimulatorTransportAdapter
  - Files: hardware/core/tests/transport_simulation_test.py pattern

Phase 3: Manager Integration Tests
  - Test watchdog in DeviceManager._scan() loop
  - Mock devices with error conditions
  - Verify scan loop error handling
  - Track health check execution
  - Create new test file: extraction_line/tests/device_watchdog_test.py

Phase 4: State Machine Tests
  - Verify executor state machine unaffected
  - Test telemetry capture of health operations
  - Verify no interference with active runs
  - Use executor_state_machine_test.py pattern

RISK AREAS FOR WATCHDOG IMPLEMENTATION:

HIGH RISK:
  1. DeviceManager._scan() loop - no error handling, must not break
  2. Device.ask() - existing retry logic, must coordinate
  3. Bootstrap lifecycle - watchdog init timing
  4. State machine transitions - executor control flow

MEDIUM RISK:
  1. Telemetry system - must log operations correctly
  2. Device.lock_scan() - contention with health checks
  3. Communication errors - must distinguish from operational errors

LOW RISK:
  1. Configuration loading - watchdog config isolated
  2. UI display - health status independent of operations
  3. DVC/database - watchdog doesn't save data

MIGRATION PATH:

Baseline (Current):
  - Device has ask() with built-in retries (2-3 attempts, 25ms)
  - EthernetCommunicator has error_mode flag
  - SerialCommunicator has retry_on_exception flag
  - No device health tracking
  - No periodic heartbeats

Phase 1 (Add Watchdog):
  - Add heartbeat_period to BaseCoreDevice
  - Add last_heartbeat_time tracking
  - Add health status enum (OK, STALE, UNREACHABLE)
  - Tests: has_communicator_test.py pattern

Phase 2 (Integration):
  - Add watchdog execution to DeviceManager._scan()
  - Coordinate with existing retry logic
  - Use simulation adapter for testing
  - Tests: transport_simulation_test.py pattern

Phase 3 (Recovery):
  - Add recovery decision logic
  - Trigger device reset/reinit on failure
  - Coordinate with executor state machine
  - Tests: executor_state_machine_test.py pattern

KEY INSIGHTS:

1. Error Path Coverage Gap:
   - Retry logic exists but not extensively tested
   - No tests for sustained disconnections
   - No device health aggregation tests
   - No recovery decision tests

2. Test Isolation is Critical:
   - Hardware tests can be skipped (traits optional)
   - Core tests have zero dependencies
   - Simulation layer avoids real hardware
   - This enables safe local development

3. Integration Points are Few:
   - DeviceManager._scan() is main entry point
   - BaseCoreDevice.ask() is communication point
   - TelemetryContext captures operations
   - Executor state machine tracks overall state

4. Minimal Footprint Possible:
   - Heartbeat logic can live in BaseCoreDevice
   - Manager integration is 5-10 lines
   - Tests can use existing simulator framework
   - No new dependencies required

5. Regression Risk Mitigated By:
   - Comprehensive existing test suite
   - Simulation framework for isolation
   - Lightweight harness patterns
   - Clear separation of concerns
